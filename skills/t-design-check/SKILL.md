# 技术设计质量检查

## Runtime Dependencies

以下路径属于目标项目运行时依赖，不是本 skill 自带资源：
- `spec/`
- `docs/`
- `.ai/`

本 skill 内部引用的插件资源应保持在 `skills/`、`agents/`、`protocols/` 下；外部路径仅表示目标项目仓库中的运行时文件。

## 目标
- 评估技术设计文档的可实施性、完整性与一致性。
- 给出可复查的 100 分量化结果。
- 输出 P0/P1/P2 修复清单。
- 为后续任务拆解提供明确的设计质量门禁结论。

## 使用方式
```bash
/t-design-check [方案名称]
```

| 参数 | 说明 |
|---|---|
| `[feature]` | 方案名称（必填） |

## 输入范围
- 设计文档：`.ai/design/[feature].md`
- 需求来源：`docs/user-stories/**/*.md`、`docs/prd/**/*.md`
- 规范来源：
  - `spec/core/environment-and-testing-guide.md`
  - `spec/core/quality.md`
  - `spec/backend/development.md`
  - `spec/frontend/development.md`
  - `AGENTS.md`

## 执行流程
1. 校验设计文档是否存在。
2. 从设计文档提取引用的用户故事、PRD、接口、数据库变更、前端范围、测试策略。
3. 核对设计文档与需求来源的一致性。
4. 核对设计文档与项目规范的一致性。
5. 检查 API 接口设计完整性。
6. 检查数据库设计完整性与简洁性。
7. 检查前端设计完整性。
8. 检查测试策略与风险说明。
9. 生成评分与问题清单。
10. 写入报告：`.ai/quality/design-check-[feature]-[YYYYMMDD-HHMMSS].md`。

## 评分体系（总分 100）

| 维度 | 分值 | 说明 |
|---|---|---|
| 需求追溯性 | 20 | 与用户故事、PRD 的目标、边界、术语一致 |
| 现有实现分析准确性 | 15 | 引用了真实路径，复用点与受影响范围明确 |
| API 设计完整性 | 20 | 接口清单、字段、错误响应、权限和路径规范完整 |
| 数据库设计完整性 | 20 | 表/字段、类型、约束、索引、迁移策略明确 |
| 前端设计完整性 | 10 | 页面/组件、线框说明、状态与数据依赖明确 |
| 测试与验收策略 | 10 | 后端/前端/Demo 测试入口和主验收路径明确 |
| 文档规范与假设显式化 | 5 | 结构稳定、假设清晰、风险可追溯 |

## 关键扣分规则

### P0（阻塞）
- 设计文档不存在
- 功能涉及后端接口，但缺少 API 接口设计章节
- 功能涉及数据库变更，但缺少数据库设计章节
- 功能涉及前端，但缺少页面/交互说明
- 设计目标与 PRD/用户故事明显冲突
- 设计文档使用不存在的仓库路径作为实现依据

### P1（重要）
- API 路径、权限模型、关键字段命名与仓库规范严重不一致
- 数据库字段类型、约束或迁移策略不明确，无法指导迁移
- 数据库设计明显过度，默认引入不必要的审计表或复杂审计机制
- 前端页面仅写"更新页面"，缺少页面区域、交互或状态说明
- 测试策略缺少实际入口，无法支撑后续任务拆解

### P2（优化）
- 可复用点说明不够聚焦
- 风险与待确认事项不够具体
- 示例 JSON 与字段表存在轻微重复或表述可压缩

## API 检查要求
- 接口清单至少包含：方法、路径、用途、权限/身份、调用方
- 关键接口至少包含：请求字段、响应字段、错误响应或状态码
- 路径参数占位符使用 camelCase，例如 `{realmId}`、`{userId}`
- 明确新增 DTO 与复用 DTO 的边界

## 数据库检查要求
- 涉及数据库时，至少给出表/字段变更清单
- 每张关键表至少说明：字段、类型、主键、唯一约束、必要索引、外键、时间字段
- 必须说明迁移策略：新增表 / 加字段 / 改名 / 是否回填 / 兼容性影响
- 默认要求设计简洁，不因潜在未来需求预埋复杂审计方案

## 前端检查要求
- 涉及前端时，至少给出页面/组件清单
- 必须有页面线框说明：页面区域、主要交互、关键状态、数据来源或依赖
- 必须说明与现有前端模式的承接关系；若不适用，应明确写原因
- 不要求前端章节单独展开 API 契约；整体 API 设计章节存在即可

## 报告要求
报告必须包含：
- 总分、等级、是否通过设计验收
- 每个维度得分与扣分证据
- P0/P1/P2 问题列表（含文件定位）
- 明确修复步骤

等级建议：
- 90-100：优秀，可进入后续拆解
- 75-89：良好，建议先修 P1
- 60-74：需改进，必须修 P0
- <60：不合格，建议重写关键章节

## 错误处理

| 错误码 | 触发条件 | 用户可见提示 | 恢复动作 |
|---|---|---|---|
| `DESIGN_DOC_MISSING` | 设计文档不存在 | 未找到设计文档 | 先运行 `/t-design [feature]` |
| `DESIGN_DOC_INVALID` | 设计文档缺少标题或主要章节结构 | 设计文档结构不完整 | 按模板补齐章节后重试 |
| `REQUIREMENT_SOURCE_MISSING` | 无法定位任何关联的用户故事或 PRD | 未找到可追溯的需求来源 | 在设计文档中补充引用后重试 |
| `REPORT_WRITE_FAILED` | 质量报告写入失败 | 无法写入检查报告 | 检查 `.ai/quality/` 目录权限后重试 |

## 示例

```bash
/t-design-check realm-user-rbac
```

输出：
```text
总分: 91/100 (优秀，可进入后续拆解)

需求追溯性: 19/20 (-1: 缺少一个用户故事来源)
现有实现分析准确性: 15/15
API 设计完整性: 18/20 (-2: 缺少 409 错误响应说明)
数据库设计完整性: 20/20
前端设计完整性: 9/10 (-1: 空态说明不足)
测试与验收策略: 10/10
文档规范与假设显式化: 5/5

P1 问题:
1. `4.2 API 接口设计` 缺少冲突场景错误响应说明

修复步骤:
1. 在关键写接口下补充 409/422 等业务错误响应
```

## 质量门禁
- 分项分值之和必须等于 100。
- 每个扣分项必须有文件定位。
- 结论必须可追溯到证据。
