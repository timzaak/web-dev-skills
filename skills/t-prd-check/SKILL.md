# PRD & User Story Quality Check

## Runtime Dependencies

以下路径属于目标项目运行时依赖，不是本 skill 自带资源：
- `spec/`
- `docs/`
- `.ai/`

本 skill 内部引用的插件资源应保持在 `skills/`、`agents/`、`protocols/` 下；外部路径仅表示目标项目仓库中的运行时文件。

## 目标
- 验证 PRD 文档完整性和规范性
- 评估用户故事质量（INVEST 原则、GWT 格式）
- 检查 PRD 与用户故事的一致性
- 检查 PRD / 用户故事是否错误混入接口、建表、schema 等实现细节
- 输出量化评分和修复清单

## 使用方式
```bash
/t-prd-check [feature-name]
/t-prd-check --all
```

## 输入范围
- PRD 文档: `docs/prd/**/*.md`（排除 `00-index.md`）
- 用户故事: `docs/user-stories/**/*.md`
- 排除文件: `00-index.md`, `_roles.md`, `_README.md`, `client-app-settings.md`, `builtin_protection.md`

## 执行流程

### 1. 确定检查范围
- 解析命令参数，确定单功能或全量检查
- 使用 `Glob` 发现目标文件并排除特殊文件

### 2. 读取角色定义
- 读取 `docs/user-stories/_roles.md`
- 解析角色名称和技术标识，用于校验故事中的角色引用

### 3. PRD 检查

#### 3.1 基础章节检查（权重 50%）

使用 `Grep` 检查必需章节是否存在：

| 检查项 | 模式 | 权重 |
|--------|------|------|
| 标题元数据 | `^\*\*创建时间\*\*\|^\*\*状态\*\*` | 5 |
| 相关用户故事 | `^## 1\. 相关用户故事` | 10 |
| 范围界定 | `^## 2\. 范围界定` | 20 |
| 需求概述 | `^## 3\. 需求概述` | 10 |
| 当前实现状态 | `^## 4\. 当前实现状态` | 5 |
| 功能需求 | `^## 5\. 功能需求` | 10 |
| API 相关约束 | `^## [0-9]+\. API 相关约束` | 10 |
| 前端/交互约束 | `^## [0-9]+\. 前端/交互约束` | 5 |
| 技术设计承接 | `^## [0-9]+\. 技术设计承接` | 5 |
| 相关文件索引 | `^## 9\. 相关文件索引` | 5 |
| 参考资料 | `^## (1[0-2]|10|11|12)\. 参考资料` | 5 |

#### 3.2 用户故事引用检查（权重 20%）

| 检查项 | 验证方式 | 权重 |
|--------|----------|------|
| 用户故事链接有效性 | 检查引用链接是否存在 | 10 |
| 优先级标注 | 是否有 P0/P1/P2 标注 | 5 |
| 优先级汇总表 | 是否有优先级汇总章节 | 5 |

#### 3.3 PRD 分层与禁止内容检查（权重 30%）

使用 `Read` 和 `Grep` 检查 PRD 是否只保留产品语义，不复制实现细节：

| 检查项 | 验证内容 | 扣分 |
|--------|----------|------|
| 接口说明引用 | API 章节包含相关接口说明或设计文档引用 | 0/-5 |
| 能力边界 | API 章节描述能力范围、访问控制、边界原则 | 0/-5 |
| 禁止端点明细 | 不出现 `GET /api`、`POST /api`、`PUT /api`、`DELETE /api` | 0/-10 |
| 禁止 schema 表格 | 不出现请求/响应字段表或参数表 | 0/-5 |
| 禁止数据库设计 | 不出现 `CREATE TABLE`、`ALTER TABLE`、`migration`、`数据库表` | 0/-5 |
| 禁止代码类型示例 | 不出现 `pub struct`、`interface Xxx`、技术代码块 | 0/-5 |

### 4. 用户故事检查

#### 4.1 故事结构检查（权重 30%）

| 检查项 | 验证方式 | 权重 |
|--------|----------|------|
| 用户故事格式 | 检查"作为"、"我希望"、"从而"三要素 | 10 |
| GWT 验收标准 | 检查 Given/When/Then 格式 | 15 |
| 场景完整性 | 每个故事至少 1 个成功场景 + 关键失败场景 | 5 |

#### 4.2 INVEST 原则检查（权重 40%）
- Independent: 故事是否独立，无明显依赖
- Negotiable: 描述意图而非具体实现
- Valuable: 明确说明用户价值
- Estimable: 可评估工作量
- Small: 范围适中，可独立完成
- Testable: 有明确的验收标准

#### 4.3 禁止内容检测（权重 30%）

使用 `Grep` 搜索禁止内容：

| 类型 | 检查模式 | 扣分 |
|------|----------|------|
| 技术实现章节 | `【技术实现】` | -10 |
| 技术视角 | `系统支持\|后端需要\|使用 Redis\|数据库表` | -5 |
| 技术表格 | `Provider 表\|provider 表\|oauth_provider_config` | -5 |
| 实现细节 | `验证码 60 秒\|调用.*接口\|前端使用\|API 路径\|新增字段\|数据库迁移` | -5 |
| 代码示例 | ```(javascript\|typescript\|rust\|python\|go\|java)` | -5 |
| 无用户/价值 | `增加按钮\|实现导出\|优化流程` | -5 |
| 模糊表述 | `良好体验\|合理处理\|适当提示` | -3 |
| API 文档 | `GET /api\|POST /api\|HTTP 状态码` | -5 |
| 数据设计 | `CREATE TABLE\|ALTER TABLE\|索引\|建表` | -5 |

#### 4.4 质量门禁检查（新文档）
- 提取文档创建时间，识别 <7 天新文档
- 新文档必须满足：
  - 无【技术实现】章节
  - 无技术表格
  - 无代码示例
  - 无 API 文档
  - 无建表、迁移、schema、类型定义等实现细节
- 违反任一规则 → 记录为 P0

### 5. 一致性检查
- 检查 PRD 中的用户故事链接是否有效
- 比较 PRD 与用户故事中的优先级是否一致
- 校验用户故事中的角色是否存在于 `_roles.md`

### 6. 评分计算
- `PRD Score = 基础章节得分 + 用户故事引用得分 - PRD分层扣分`
- `User Story Score = 故事结构得分 + INVEST 得分 - 禁止内容扣分 - 质量门禁扣分`
- `Consistency Score = 用户故事链接有效性 × 5 + 优先级一致性 × 5 + 角色引用正确性 × 5`
- `Total Score = (PRD Score × 50%) + (User Story Score × 50%) + Consistency Score`

### 7. 问题分级

| 级别 | 影响 | 示例 |
|------|------|------|
| P0 | 阻塞实现 | 缺少核心 PRD 章节、用户故事无验收标准、新文档含【技术实现】章节、PRD 含端点/schema/建表等实现细节 |
| P1 | 影响质量 | PRD 缺少必要的接口说明引用、模糊验收标准、INVEST 原则违反、旧文档含技术表格 |
| P2 | 优化建议 | 前端/交互约束不完整、格式问题 |

### 8. 输出要求
- 控制台摘要：总分、评级、P0/P1/P2 统计、质量门禁结果、PRD 分层检查结果、前 3 个关键问题、报告路径
- 详细报告文件：`.ai/quality/prd-check-[YYYYMMDD-HHMMSS].md`
- 报告应包含：总体评估、质量门禁、PRD 分层检查、各功能详细得分、P0/P1/P2 问题、优先修复建议

### 9. 失败处理
- 未找到文档：提示检查功能名称或改用 `--all`
- 文件解析错误：记录错误并继续其他文件
- 报告目录不存在：使用 `Bash` 创建 `.ai/quality/`
