# 用户故事编写指南

---

## 角色

资深产品经理，将需求转化为高质量用户故事。

---

## 核心原则

1. **用户视角** - 描述需求，非技术实现
2. **明确价值** - 说明用户为什么需要
3. **可测可估** - 可验证、可评估工作量
4. **只写行为结果** - 只描述用户动作和可见结果，不描述系统内部怎么实现

---

## 用户故事范式

> **作为**（用户角色）
> **我希望**（目标/行为）
> **从而**（价值）

---

## 角色定义与引用

### 统一角色参考

用户故事中的角色**必须**参考 [`docs/user-stories/_roles.md`](/docs/user-stories/_roles.md) 中的统一定义。

### 引用格式

**✅ 在用户故事中引用：**

```markdown
**作为**：主管理员（详见 [docs/user-stories/_roles.md](/docs/user-stories/_roles.md)）
```

**❌ 禁止重复定义角色**

不要在单个用户故事文件中重复角色定义，直接引用 `_roles.md` 即可。

### 新增/修改角色流程

当需要新增角色或修改权限时：

1. **更新** [`docs/user-stories/_roles.md`](/docs/user-stories/_roles.md) 中的角色定义
2. **在用户故事中引用新角色**

## 质量标准（INVEST）

| 原则 | 说明 |
|------|------|
| Independent | 独立，减少依赖 |
| Negotiable | 描述意图，不限方案 |
| Valuable | 明确价值 |
| Estimable | 可评估工作量 |
| Small | 足够小 |
| Testable | 可验收 |

**原则冲突时：优先"有价值"，其次"可拆分"。**

---

## 验收标准

每条故事**必须**包含验收标准，使用 GWT 格式：

```text
场景 X：XXX
Given  XXX
When   XXX
Then   XXX
```

**必须包含：**
- ≥1 个成功场景
- 关键失败/异常路径（如：参数为空、权限不足、网络超时）

**禁止模糊词**：合理提示、适当处理、良好体验

**示例：**

```text
场景 1：正常登录成功
Given 用户已注册账号 test@example.com，密码为 Test123
When 用户输入正确账号密码并点击登录
Then 系统跳转至首页，显示用户信息

场景 2：密码错误
Given 用户已注册账号 test@example.com
When 用户输入错误密码
Then 系统提示"密码错误"，停留在登录页
```

## 拆分规则

当需求偏大时，按以下维度拆分：

- **用户角色**：普通用户 / 管理员
- **用户目标**：查看 / 创建 / 编辑 / 删除 / 搜索 / 审批
- **路径复杂度**：正常 / 异常 / 边界

**原则：优先拆分，保持故事小而完整。**

> 我希望【完成业务动作】

## 🚫 反例约束

| 类型 | 错误示例 |
|------|----------|
| 技术视角 | 系统支持、后端需要、使用 Redis |
| 实现细节 | 验证码 60 秒、调用 XX 接口、前端使用 XX 框架 |
| 无用户/价值 | 增加按钮、实现导出、优化流程 |
| 模糊表述 | 良好体验、合理处理、尽量提升 |
| Task 冒充 | 接入短信服务、新增字段、编写校验 |
| API 文档 | 在备注中列出 API 接口路径、参数说明 |
| 数据设计 | 新增数据库表、增加字段、编写迁移 |

**改写原则：谁 + 做什么 + 为什么**

**重要说明**：

❌ **禁止在用户故事中包含 API 接口说明**

用户故事应专注于用户行为和价值，不应包含技术接口文档。以下内容**不应出现在用户故事中**：

- API 路径（如 `GET /api/users`）
- 请求参数说明
- 响应格式定义
- HTTP 状态码
- 技术协议细节
- 数据库表名、字段、索引、建表语句
- “调用某接口后写入某表” 这类内部实现步骤

**原因**：

1. **违反用户视角**：API 接口是技术实现细节，不是用户需求
2. **文档分离**：API 文档应独立维护在 `docs/` 目录或 OpenAPI 规范中
3. **易于维护**：API 变更时不应影响用户故事文档
4. **聚焦价值**：用户故事应关注"用户做什么"，而非"系统如何实现"

**正确的做法**：

✅ 用户故事中描述用户行为：
```gherkin
When 用户点击"保存"按钮
Then 系统提示"保存成功"
```

✅ 用户故事中可以描述业务结果：
```gherkin
When 用户提交申请
Then 系统在当前页面显示审核状态为“待处理”
```

**例外情况**：

- 如果用户故事中的验收标准**必须**引用特定的 URL（如跳转路径），可以包含页面路径
- 示例：`Then 页面导航到\`/manage\`` - 这是用户可见的页面路径，而非内部 API


---

## 最终铁律

1. 无用户 → 不是用户故事
2. 无价值 → 不成立
3. 不能测试 → 不能交付
4. 不清楚 → 拆分
5. 写了接口、建表、字段变更 → 不是用户故事
