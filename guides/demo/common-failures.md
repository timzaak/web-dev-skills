# 常见稳定性问题

适用场景：

- `strict mode violation`
- 元素不可点击或不可见
- 测试超时
- 本地通过、CI 失败

## 处理顺序

1. 先确认是测试稳定性问题，不是服务异常。
2. 再按失败类型做最小修复，不要同时改等待、选择器、业务断言三类逻辑。

## 问题到动作

### strict mode violation

- 缩小选择器范围
- 使用更精确的 role、文本或 `data-testid`
- 只有在语义明确时才用 `first()` / `nth()`

### 元素不可交互

- 先确认元素真的可见且未被遮挡
- 需要时使用 `scrollIntoViewIfNeeded()`
- `force: true` 只作为最后手段

### 测试超时

- 优先补业务断言或显式等待条件
- 不要先加大超时掩盖竞态
- 若确为慢请求，再调整局部超时

### CI-only failure

- 检查数据隔离和时间戳命名
- 检查是否依赖本地特有状态或浏览器节奏
- 确认选择器是否过度依赖布局、动画或瞬时文案

## 完成判定

- 修复后的等待条件可解释
- 没有新增固定 sleep
- 本地验证通过后再进入完整命令链路
