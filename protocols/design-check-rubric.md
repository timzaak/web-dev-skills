# Design Check Rubric

定义 `t-design-check` 的统一评分和判定标准。

## Scoring

总分 100：

| 维度 | 分值 | 说明 |
|---|---:|---|
| 需求追溯性 | 20 | 与用户故事、PRD 的目标、边界、术语一致 |
| 现有实现分析准确性 | 15 | 使用真实路径，复用点与受影响范围明确 |
| API 设计完整性 | 20 | 接口清单、字段、错误响应、权限和路径规范完整 |
| 数据库设计完整性 | 20 | 表/字段、类型、约束、索引、迁移策略明确 |
| 前端设计完整性 | 10 | 页面/组件、线框说明、状态与数据依赖明确 |
| 测试与验收策略 | 10 | 后端/前端/Demo 测试入口和主验收路径明确 |
| 文档规范与假设显式化 | 5 | 结构稳定、假设清晰、风险可追溯 |

## Severity Rules

### P0

- 设计文档不存在
- 涉及后端接口但缺少 API 设计
- 涉及数据库变更但缺少数据库设计
- 涉及前端但缺少页面/交互说明
- 设计目标与 PRD/用户故事明显冲突
- 使用不存在的仓库路径作为依据

### P1

- API 路径、权限模型、关键字段命名与仓库规范严重不一致
- 数据库字段类型、约束或迁移策略不明确
- 数据库设计明显过度
- 前端页面描述过于抽象，缺少页面区域、交互或状态
- 测试策略缺少实际入口

### P2

- 可复用点说明不够聚焦
- 风险与待确认事项不够具体
- 示例与字段表轻微重复，可压缩

## Detailed Checks

API：

- 至少包含方法、路径、用途、权限/身份、调用方
- 关键接口包含请求字段、响应字段、错误响应或状态码
- 路径参数占位符使用 camelCase
- 明确新增 DTO 与复用 DTO 的边界

数据库：

- 涉及时给出表/字段变更清单
- 关键表说明字段、类型、主键、唯一约束、必要索引、外键、时间字段
- 说明迁移策略和兼容性影响
- 默认保持简洁，不预埋不必要复杂审计方案

前端：

- 涉及时给出页面/组件清单
- 给出页面区域、主要交互、关键状态、数据来源或依赖
- 说明与现有前端模式的承接关系

## Report Requirements

报告必须包含：

- 总分、等级、是否通过设计验收
- 每个维度得分与证据
- P0/P1/P2 问题列表
- 明确修复步骤

等级建议：

- `90-100`: 优秀，可进入后续拆解
- `75-89`: 良好，建议先修 P1
- `60-74`: 需改进，必须修 P0
- `<60`: 不合格，建议重写关键章节
